# 2024-wu-4dgs · 4D Gaussian Splatting for Real-Time Dynamic Scene Rendering

## 一句话问题
如何在保留 3D Gaussian Splatting 实时渲染能力的前提下,**整体表达**(非逐帧)动态场景的运动与形变,并保持高分辨率下的实时帧率?

## 方法核心(3~5 条要点)
1. **整体表示,非逐帧 3DGS**:不把动态场景拆成 N 个独立 3DGS,而是把 3D 高斯 + 4D 神经体素作为统一表示。
2. **HexPlane 启发的分解式神经体素编码**:把 4D 时空体素(空间 × 时间)分解为 6 个特征平面(类似 T HexPlane / X-Y / X-T / Y-T ...)的乘积组合,大幅降低 4D voxel grid 显存与查询复杂度。
3. **可学习变形场**:canonical 3D 高斯在每个时间戳 t 由一个轻量级 MLP 预测形变(位姿 / 旋转 / 尺度 / 不透明度),所以本质是 canonical + deformation 范式。
4. **邻居高斯结构化耦合**:变形场额外通过空间-时间结构编码把"相邻高斯"连接,使运动/形变在邻域内连贯(避免孤立形变）。
5. **直接用 3D Gaussian Splatting 光栅化器渲染**:每帧重建出的 3D 高斯直接走标准 splatting rasterizer,无需发明新渲染核,所以天然继承 3DGS 的实时性。
6. **数据集自监督监督**:用光度 L1 + D-SSIM 损失 + 正则化在多视角视频序列上自监督训练,无需 GT 几何 / 运动。

## 与本调研主线的关系
- **主线对标(必查)**:`docs/00-goal.md` §"第一块调研 · 必须覆盖"明确把 4DGS 原论文列为对标主线之一;`§"对标主线"`列在第一位。
- **借鉴价值**:
  1. HexPlane 分解编码是后续几乎所有 4DGS / 4DGS-1K(-lite) 加速路线的**共同前置依赖**——调研时需把 4DGS 的 HexPlane 视为"原始 SOTA 训练路径",所有衍生加速都必须能落在 HexPlane 输出之上。
  2. canonical + deformation 范式意味着 4DGS 在推理时的"额外代价"集中在 deformation MLP 上,这恰好是后续做**时间复用 / mask 共享**的天然分界点。
- **不可作为移动端最终方案**:82 FPS @ RTX 3090(桌面) vs 目标 Adreno 8 Gen 4(手机);论文未给出移动端数字。**反例基准**:本论文是移动端"对比上限",不是"最终方案"。
- **对采集端反推**(00-goal.md §"必须覆盖"的"对采集端的反推"项):4DGS 论文使用 D-NeRF synthetic(单目视频序列)+ Plenoptic Video Dataset;采集端约束并不直接对应"相机阵列 + 高精度 SfM"那条主线;但数据集假设本身说明 4DGS 训练对"高质量重建 + 多视角同步"有相当要求。
